# 1. サービス概要

- 食材をタップするだけで、10分以内に作れる「お弁当向け副菜」をAIが提案するWebサービスです。
- 平日の朝に「あと1品どうしよう」と迷う人の、決断疲れを解消します。
- レシピを“検索する”時間をなくし、“決めてもらえる”体験を提供します。

---

# 2. このアイデアはどこから生まれたか

## 2-1. きっかけとなった体験・感情

平日の朝、出勤前にお弁当を準備しているとき。
冷蔵庫に食材はあるのに、「何を作るか」が決まらず手が止まる。
レシピサイトを開いても情報が多すぎて選べない。
時間がないのにスマホを眺めている自分に焦りを感じました。
特に「あと1品どうしよう」という副菜で毎回止まることが負担になっていました。

---

## 2-2. なぜそれが「気になった」のか

- 自分は効率を重視する性格で、無駄な時間が強いストレスになる
- 朝は気持ちに余裕を持ちたいという価値観がある
- 過去に時間に追われる生活に疲れた経験がある

料理そのものよりも、
“決める時間”にエネルギーを消耗していることが問題だと感じました。

# 7. このアプリで実現すること

### MVPで作る機能

- 食材タップ選択機能（最大3つ）
- AI副菜提案機能（10分以内制約付き）
- 再提案ボタン

## 7-1. MVP設計に関する補足検討事項
■ 食材入力の粒度について

食材は「にんじん」「卵」などの単体食材レベルで選択できる設計とします。
理由は、入力負担を最小限にし、「迷わず即決する体験」を守るためです。
具体的には以下を検討しています。

- よく使う食材を上位表示
- ワンタップで選択できるUI
- 最大3食材までに制限（選択肢過多を防ぐ）
- 食材未選択でも提案される「即決モード」
入力で疲れさせないことを最優先とします。

---

■ 提案が好みに合わなかった場合の対応

副菜提案が合わない場合に再び検索へ戻ってしまうと、本来解決したい「決断疲れ」が再発します。
そのためMVPでは、
同一食材で即時再提案できる「再生成ボタン」を実装します。

将来的には、
- 味付け傾向の学習
- 「あっさり／こってり」などの気分選択
- 嫌いな食材・味付けの事前登録
により、再提案回数自体を減らす設計を目指します。

---

■ 急いでいるユーザーへの配慮

朝は時間的制約が強いため、
AIは内部的に複数案を同時生成し、
UI上ではワンタップで切り替え可能にする設計を検討しています。

再リクエストを発生させず、
“すぐ別案を見る”体験を提供することで朝のストレスを最小化します。

---

■ 「10分以内」の制約への対策

生成AIは時間概念を正確に理解しない可能性があるため、
アプリ側で以下の制御を行います。

- 工程数を最大5ステップ以内に制限
- 長時間加熱（煮込み等）を禁止
- 電子レンジ・フライパン調理のみ許可
- 出力形式をJSON化し、工程数を検証
AI任せにせず、アプリ側でも制御する設計とします。

# 8. このアプリの懸念点とその対策

### 懸念点①

- 何が問題になりそうか：
    AIレシピの品質が安定しない可能性
    
- なぜそう思うか：
    生成AIは出力が揺れるため
    
- そのために考えている対策：
    - 出力形式をJSON固定
    - 工程数を5以内に制限
    - 長時間加熱を禁止
    - 「家庭調理可能」と条件明示
    - プロンプト改善と検証を繰り返す

# 10. 技術スタック

## 10-1. 使用予定の技術

- フレームワーク：Ruby on Rails 7
- DB：PostgreSQL
- デプロイ先：Render
- 使用予定ライブラリ：OpenAI API / Hotwire / TailwindCSS

---

## 10-2. 技術選定の理由

- Railsは学習内容と一致し、開発速度が速い
- HotwireでJSを最小限に抑えられる
- OpenAIにより大量レシピDBを持たずに差別化可能

チャレンジしたい点：
AI出力を安定的に構造化すること

不安な点：
APIコストと出力精度の担保
